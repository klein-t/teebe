## [F4] Quick Look (context menu) — high/functional/bug

CONFIRMED. The context-menu Quick Look (FilesSection.swift:159-162) calls only `await preview.toggle(for:worktreePath:snapshot:)`; it never opens the floating window. The SwiftUI "preview" Window scene (TreebranchApp.swift:44, "Quick Look", id:"preview") is a `Window` (not WindowGroup), so it does NOT auto-open on launch and surfaces ONLY via `openWindow(id:"preview")`. grep confirms that call exists at exactly one site: RootView.swift:323, inside toggleDiffPeek() (the CHANGES spacebar path). PreviewModel.toggle (PreviewModel.swift:30-38) sets isVisible=true and resolves content, but with no openWindow the window never appears — F4 shows nothing, contradicting FEATURE_AUDIT.csv F4 ("Context menu Quick Look should show a preview of the file"). I tried to refute via alternate window-open paths: the only makeKeyAndOrderFront (QuickLookBridge.swift:52) is the native QLPreviewPanel used by the FILES *spacebar* (G1), not this SwiftUI window; PreviewPanel reacting to preview.content does not open the window. No onChange/AppKit trick opens it. Secondary inversion claim is real but conditional: after context-menu Quick Look (isVisible=true, no window), a subsequent CHANGES spacebar hits `if preview.isVisible` (RootView.swift:314) → close()+dismissWindow instead of open, a visible no-op needing two presses; this corrupts shared preview state. Minor correction to the prior auditor: the FILES spacebar path (presentQuickLook, RootView.swift:330-336) uses the native QLPreviewPanel and is independent of preview.isVisible, so it is NOT inverted — only the CHANGES diff-peek toggle is. Severity high/functional is justified (feature entirely non-functional + poisons diff-peek state), not inflated. Fix: inject @Environment(\.openWindow) and call openWindow(id:"preview") after preview.toggle in the context-menu action (or hoist to RootView and mirror toggleDiffPeek).

## [J2] Record write activity — med/functional/bug

Confirmed via my own read. The FSEvents callback (WorktreeModel.swift:138-144) ignores the affected paths (`_`) and unconditionally calls activityMonitor.recordActivity for ANY change in the watched worktree; there is no origin token or self-write filter. FSEventsWatcher (FSEventsWatcher.swift:37-100) does zero path filtering and watches [worktreePath] with kFSEventStreamCreateFlagFileEvents, so any write under the worktree root flushes the handler. recordActivity has no notion of who wrote (RepoGitQueue.swift:103-118). I traced which teebe-originated UI mutations provably write INTO the watched tree and thus self-record: discard/discardUntracked (WorktreeModel.swift:384-386 -> git checkout/clean rewrites working files), trash (390-392 -> moveToTrash removes a worktree file), and file ops new/rename/duplicate (AppModel.swift:107-126 -> FileOps mutate worktree, then runFileOp refresh at 134-141). These fire FSEvents in the watched root regardless of worktree type, so isBusy becomes true for the user's own action. Consequences are both reachable: (a) the next guarded op shows '⚠︎ ...an agent may be active' (RootView.swift:365 <- mutation.worktreeBusy <- isBusy at WorktreeModel.swift:404-407, set at requestDiscard/requestTrash 363/371) within the 5s window; (b) the green LiveDot pulses on the worktree row (WorktreesSection.swift:48,118 <- info.isLive <- Selector/AppModel refreshWorktreeInfo isBusy, SelectorModel.swift:78 / AppModel.swift:78). PARTIAL REFUTATION of the prior auditor's evidence: their stage/unstage/commit citations are imprecise for LINKED worktrees, whose gitdir (index/HEAD/refs) lives under repo/.git/worktrees/<name>, OUTSIDE the watched worktree path, so index-only ops may not fire FSEvents there (they reliably do only on the primary worktree where .git is inside the root). But this does not rescue the verdict: discard, trash, and new/rename/duplicate are sufficient, reachable false-positive sources on any worktree. Verdict bug, severity med (genuine false positive undermining the product's headline 'agent activity' signal; but fail-safe — warns too much never too little, no data loss, guard still protects), type functional (incorrect activity attribution; surfaces as UX but the root cause is logic that conflates self vs external writes). Fix per prior hint (origin token / ignore-window around teebe's own writes) is valid.

## Cross-cutting / extra defects

### Live/sync state never auto-updates; only repo-select and the manual refresh button recompute it (med/ux)

The WORKTREES list's sync arrows (C3), change counts, and live dots (C4) are computed once at repo selection and then frozen until the user manually clicks refresh. After committing, staging, or any FSEvents activity, the per-row ahead/behind and live indicators stay stale even though the active worktree's CHANGES/FILES (driven by the separate WorktreeModel.refresh) update live. This makes the at-a-glance divergence/live cues unreliable.

**Fix:** Call refreshWorktreeInfo() after selectWorktree and after mutations/commits, and/or recompute the active worktree's info from the FSEvents refresh path (debounced) so the list reflects current state without manual refresh.

### Commit ignores the unstaged half of a partially-staged file (low/functional)

If a user has a file that is partially staged (some hunks staged, more edits made afterward), pressing Commit commits only the previously-staged version and silently leaves the newer edits out of the commit, even though the file is counted in the 'Commit N changes' badge. This matches the literal spec wording ('stage everything not staged') but is a surprising correctness gap for the user.

**Fix:** Stage the working side of every non-untracked change too (e.g. always `git add -- <path>` for files whose worktreeStatus != .unmodified), or document that staged files commit as-staged.

### Context-menu Quick Look desyncs the diff-peek toggle (isVisible set with no window) (med/functional)

After invoking context-menu Quick Look on a file (which shows nothing — see G5), preview.isVisible is left true. The next space press on a CHANGES row is consumed as a 'close' instead of opening the diff peek, so the user must press space twice. This is the same root cause as G5 but produces a distinct, surprising failure on the otherwise-working G2 path.

**Fix:** Have the context-menu Quick Look open the preview window (openWindow id:"preview") so isVisible and the window stay in sync, or stop using the shared PreviewModel.isVisible for the context-menu path.
